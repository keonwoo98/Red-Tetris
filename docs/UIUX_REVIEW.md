# Adversarial Review — Red Tetris UI/UX Redesign Spec

Verdict: the spec is **constraint-compliant and largely feasible**. Its "reality checks" against the codebase are accurate (verified: per-cell `0 0 11px` glow, `--cell: clamp(15px,2.3vw,30px)`, `body::before/::after` over everything, `LOCK_DELAY_FRAMES=1`, `PENALTY=8` indestructible, `listeners.ts:29` discards `fromId/fromName`, win is server-side `last-standing`). No hard-constraint violations found. The issues below are about **feasibility risk, scope mis-ranking, and a few correctness traps** that will bite an engineer if implemented as written.

## (a) Constraint Compliance — Issues

- **[PASS] No Canvas/SVG/table/jQuery/charting.** Every proposed effect (confetti, pillars, crumble, ghost-fields, garbage meter, countdown rings) is `div` + CSS keyframes/transforms/clip-path/gradients. Confirmed compliant. The confetti "~24 divs" is the only place that smells like a chart library — it isn't; it's plain divs. Keep it ≤24 to stay cheap.
- **[PASS — but add a guard] Opponent info limit.** §5a ghost-field renders **only `spectrum: number[10]` + name + alive**, never a real board. Confirmed `OpponentDTO` exposes exactly `{id,name,alive,spectrum}` and `SpectrumUpdatePayload` carries no cells. The mini-well is a *silhouette from heights*, not a board — compliant. **FLAG for the implementer:** the mini-well must render columns purely from `spectrum[c]` height; do **not** ever try to reconstruct per-cell fill (e.g. "holes"), since spectrum has no hole data. Render solid bottom-anchored fills only.
- **[PASS] No `this` / pure engine.** Every new field (`combo`, `b2b`, `dropFx`, `lastAttack`, `armedAt`, `koSeq`) lives in Redux slices computed in reducers, and every effect is a React functional component reading selectors. Nothing pushes logic into the pure engine (`engine/*` stays untouched). Confirmed the engine has no client state. **One caution:** §3 says "compute combo/b2b/perfect in `commitLock`" — `commitLock` is in `gameSlice.ts` (reducer), **not** the engine, so this is fine. Do not move clear-type detection into `engine/lines.ts`.

## (b) Feasibility — Issues (severity)

- **[HIGH] Active-piece transform layer (§4 "enabler note") is under-costed and structurally risky.** The spec correctly marks it optional/P3, but the current `Board.tsx` bakes the active piece into `overlayForRender` (a single 200-cell grid). Splitting into a static stack grid + an absolutely-positioned `.activePieceLayer` means: (1) the layer must be positioned with `transform: translate(calc(x*var(--cell)), calc(y*var(--cell)))` — but `--cell` is a `clamp()`, and `calc()` on a clamped var works, yet **sub-cell gravity is impossible here**: the engine moves the piece a whole cell per `tick`, so `transition: transform 55ms` only smooths *cell-to-cell*, never true sub-cell fall. The "smooth gravity" framing oversells it. **Fix:** keep the framing as "smooth *interpolation between cell positions*," not "sub-cell gravity," and explicitly note `invisible` mode (which renders an empty board) must still position the active layer over the hidden stack — easy to break. Defer to last, as the spec already hints.
- **[MED] Per-row `rowSweep` clip-path collapse needs row geometry the current Board doesn't expose.** `clearFx` only carries `{lines, seq}` (now extended) — it does **not** carry *which* row indices cleared. To animate the specific cleared rows wiping then survivors settling, the component needs the cleared row indices. `lockEvent` carries the post-clear board but not which rows vanished. **Fix:** either (a) add `rows: number[]` to `clearFx` (compute in `commitLock` before `clearLines` collapses them — cheap, pure), or (b) downgrade P3's per-row sweep to a full-board horizontal sweep keyed on `clearFx.seq` (no row data needed). Recommend (a); it's a 3-line reducer change and makes the effect actually correct.
- **[MED] 200 cells each animating `box-shadow`/`filter` simultaneously can drop frames.** `lockFlash` uses `filter: brightness(2)` + animated `box-shadow` per locked cell (up to 4 cells, fine), but the Tetris pillars + whole-board `clearFlashTetris` + shake running together is the worst case. `box-shadow` and `filter` are **not** GPU-cheap to animate across many nodes. **Fix:** prefer `transform`/`opacity` for anything spanning many cells; use `filter`/`box-shadow` only on small node counts (the 4 locked cells, single popups). Add `will-change: transform` sparingly on the board frame, not on all 200 cells. This is a real 60fps risk on the 880px/low-end target.
- **[MED] Garbage rows "slide in from bottom with 1-frame red flash" (§4 Penalty IN) fights the engine model.** Penalty is flushed inside `commitLock` *before* the lock, and `applyPenalty` can also flush immediately when `current === null`. The board is replaced wholesale (immutable). There is **no animation hook** for "rows sliding in" because the new board just appears. **Fix:** trigger the *board-level* jolt/vignette/shake off a new `lastPenaltyAt` seq (the spec's `lastAttack.seq`), and animate the **whole board** (`translateY(-4px→0)` + red inset), not individual injected rows. Drop "garbage rows slide in" — it implies per-row diffing the spec doesn't provide.
- **[LOW] `transition: height` on opponent `.fill` (§5a) animating on every `spectrum:update`** is fine, but spectrum updates can arrive frequently. With 6 rivals × 10 columns = 60 simultaneously-transitioning divs, each on `height` (a layout-triggering property). **Fix:** animate `transform: scaleY()` from a bottom origin instead of `height` to stay off the layout path. Same visual, GPU-composited.
- **[LOW] Countdown sync via `startedAt` (§5e) is sound but needs a clock-skew note.** `GameStartPayload.startedAt` is a server timestamp; clients computing `startedAt - Date.now()` will be off by each client's clock skew + latency. For a 3-2-1 it's visually tolerable, but **the GO! and input-unlock must gate on local elapsed time, never on a second network event**, or fast clients start before slow ones. The spec gets this right ("start the local timer relative to `startedAt`") — just flag that input must stay locked until the local countdown completes regardless of skew.

## (c) Determinism / Spec-Win-Condition — Issues

- **[PASS, with one trap] No UI change touches the deterministic stream.** All piece advancement stays in `pieceAt(seed, pieceIndex)` inside reducers; no proposed effect calls RNG or mutates `pieceIndex`/`seed`. Confirmed. **TRAP to flag:** §3's PERFECT/ALL-CLEAR detection ("if board is fully empty") must read the board *after* `clearLines` but is purely observational — it must **not** branch piece spawning. As written it only sets `clearFx.perfect`, which is safe. Keep it observational.
- **[HIGH] Client-derived `placementOrder` (§5d) can disagree with the authoritative winner — and the spec half-admits it.** Win is server-authoritative: `game:over` carries `winnerId/winnerName/reason: 'last-standing'`. The module-level `placementOrder` appended on `opponentGameOver`/local `topOut` is an **approximation** (event ordering, the player's own topout vs. server confirmation). If the results screen ranks #1 by `placementOrder` and it disagrees with `game:over.winnerId`, the UI lies. **Fix (mandatory, not optional):** the results screen's **#1 must be `game:over.winnerId`** (authoritative); `placementOrder` may only fill ranks #2..#N for *flavor*. The spec lists the server `placement` field as "optional/flagged" — for #1 it is **not optional to defer to `winnerId`**. Also: `placementOrder` as a **module-level mutable** in `opponentsSlice.ts` is a code-smell in a Redux slice (survives across games, breaks "no `this`/pure" spirit, and StrictMode double-invoke can double-append). Put it in slice state and reset on `startGame`/`resetGame`.
- **[MED] Combo/B2B definitions risk a determinism-irrelevant but correctness bug.** `combo` resets "to −1/0 when a piece locks with `n===0`" — but `commitLock` runs on hard drop *and* gravity lock *and* the penalty-topout path. Ensure combo logic only runs on the **successful lock-and-clear branch** (after line 105's `clearLines`), not on the early `topOut` returns (lines 96–100), or a topping-out player corrupts combo state mid-death. B2B scoped to Tetris-only is a reasonable constraint workaround (no T-spin detection exists) — acceptable.

## (d) Scope / Priority — Re-ranking

The spec's P1→P3 is **mostly right** but P3 over-commits. Two corrections:

1. **P3 as written bundles the active-piece-layer refactor with lock-flash/clear/shake.** The layer is the single highest-risk item in the whole plan (see HIGH above) and is *not needed* for lock-flash, line-clear, garbage, or shake (all work on the static grid). Bundling them invites a stalled P3. **Split it out** to its own late phase (the spec already says "defer to P3 if heavy" — make that explicit: it is **not** in core P3).
2. **The `clearFx.rows` data gap (MED) must be fixed *before* P3's per-row collapse**, or P3 silently degrades to a whole-board flash (which is the very thing the spec is trying to move away from). Pull that 3-line reducer change into P3 scope.

Otherwise the P1 ranking is correct: **layout + ambient re-rank is genuinely the highest-impact, lowest-risk, zero-engine-change win** — it's pure CSS in `GameView.module.css` + `theme.css` + remounting existing components, and it delivers the "instantly looks like real Tetris" reframe. P2 (ghost-field mini-wells) is correctly second: pure render, no server change, directly attacks "multiplayer too bare," and the `fromId/fromName` data is *already in the protocol* so even the P4 attack-feedback groundwork is unblocked.
